# Keyboard Catalogue Hugo Starter

I want to have a nice of way of keep tracking the keyboard and keycaps I have. The main idea is that I not only need record keeping but also organising photos of different keycaps on different keyboards in a nice browsable way. Therefore, **[Hugo](https://gohugo.io/)** comes to mind, because it has the flexibility of local browsing and making things public.

Here, I asked **ChatGPT** to generate s theme-free Hugo skeleton for cataloguing my own keyboards, keycap sets, designers, switches, and the photos of their combinations. After a few iterations and modifications, here it is. I like the current version.

The following instructions and general usage advice are generated by ChatGPT, and you can follow or ignore some of them as you want.

If you have any questions and feedbacks, you can raise an issue and we can discuss.

## 1. Run locally

Install Hugo, then from this folder run:

```bash
hugo server --buildDrafts
```

Open the local URL Hugo prints, usually <http://localhost:1313/>.

## 2. Add a keyboard

Create a page bundle:

```bash
mkdir -p content/keyboards/my-keyboard
cp content/keyboards/example-mode-envoy/index.md content/keyboards/my-keyboard/index.md
```

Edit the front matter in `index.md`, then place photos in the same folder. Name one photo `cover.jpg`, `cover.png`, or `cover.webp` to make it the card/hero image.

### Layout terminology

The skeleton now avoids the ambiguous word `layout` by separating two ideas:

```yaml
form_factor: "65%"     # Size/class: 60%, 65%, 75%, TKL, 80%, Alice, etc.
layout_style: "WK"     # Top-case/blocker style: WK, WKL, HHKB, Tsangan, etc.
```

`form_factor` is the broad physical class of the board. `layout_style` is the blocker or top-case arrangement within that class.

For example:

```yaml
form_factor: "60%"
layout_style: "HHKB"
```

is clearer than:

```yaml
layout: "60% HHKB"
```

Older entries using `layouts: ["65%"]` or `layout: "WK"` still render, but new entries should use `form_factor` and `layout_style`.

### Multiple physical copies

Top-level keyboard fields are treated as the shared/default values for the keyboard. Rows under `copies` inherit those values unless a row overrides them.

Use this pattern when two copies are mostly the same:

```yaml
status: "owned"
acquired: "2026"
form_factor: "65%"
layout_style: "WK"
pcb: "Hotswap"
weight_material: "Copper"

copies:
  - name: "Mirage build"
    colour: "Mirage"
    finish: "Anodised"
    case: "Aluminium"
    plate: "FR4"
    switches: ["Gateron Oil King"]
  - name: "Frost build"
    colour: "Frost"
    finish: "Translucent"
    case: "Polycarbonate"
    plate: "PEI"
    switches: ["Cherry MX Black"]
```

In the rendered table, `acquired`, `form_factor`, `layout_style`, `pcb`, and `weight_material` are inherited into both rows, so you only need to write the differences.

If the form factor or layout style differs between physical copies, put the differing values in the relevant copy rows:

```yaml
form_factor: "60%"
pcb: "Solder"

copies:
  - name: "WK build"
    layout_style: "WK"
    colour: "Silver"
    plate: "Aluminium"
  - name: "HHKB build"
    layout_style: "HHKB"
    colour: "Black"
    plate: "FR4"
```

The copy table supports these fields:

```yaml
name: ""
acquired: ""
form_factor: ""         # 60%, 65%, 75%, TKL, 80%, Alice, etc.
layout_style: ""        # WK, WKL, HHKB, Tsangan, split backspace, etc.
colour: ""              # `color` also works.
finish: ""              # e.g. anodised, powdercoat, e-coat, raw, translucent.
case: ""                # e.g. aluminium, polycarbonate, acrylic.
plate: ""               # e.g. FR4, aluminium, PEI, POM.
pcb: ""                 # e.g. hotswap, solder, flex-cut.
featured_decoration: "" # e.g. brass, copper, stainless steel, PVD weight.
switches: []
notes: ""
```

Fields can still be written as a single value or a list:

```yaml
plate: ["FR4", "PEI"]
pcb: ["Solder", "Hotswap"]
```

For the cleanest copy table, only put a multi-value list at the top level if the whole list applies to every copy. If the values differ between copies, put those values inside the relevant `copies` rows.

## 3. Add a keycap set

Create a folder under `content/keycaps/` with an `index.md` and photos beside it.

To record which kits you own, use `kits`:

```yaml
keycapsets: ["GMK Botanical 2"]
profiles: ["Cherry"]
makers: ["GMK"]          # Manufacturer / vendor.
designers: ["Hazzy"]    # Keycap set designer(s).
colorways: ["Green", "Cream"]
material: ["ABS doubleshot"]
kits:
  - name: "Base"
    status: "owned"
  - name: "Novelties"
    status: "owned"
  - name: "Spacebars"
    status: "wishlist"
    notes: "Useful for split layouts."
```

For a simpler entry, strings also work:

```yaml
kits: ["Base", "Novelties"]
```

Simple string kits render as owned by default.

Use `designers` rather than `designer`, even for a single person, so the site can create browse pages under `/designers/`. The metadata panel labels it as “Designer” on the page. If a set has multiple designers, list them all:

```yaml
designers: ["Designer One", "Designer Two"]
```

## 4. Add a combination

Create a folder under `content/combinations/` with an `index.md` and the photos for that combination.

General rule: a combination page describes the **actual photographed build**, not the full master record of the keyboard or keycap set. Use the keyboard page for owned copies, cases, plates, PCBs, weights, and acquisition details. Use the keycap page for designer, manufacturer, material, and kit ownership. Use the combination page for the particular mounted result shown in the photos.

A good combination front matter has four groups:

```yaml
# 1. What is photographed?
boards: ["Mode Envoy"]
keycapsets: ["GMK Botanical 2"]

# 2. Actual keyboard configuration in this photographed build.
form_factor: "65%"
layout_style: "WK"
colour: "Mirage"
finish: "Anodised"
case: "Aluminium"
plate: "FR4"
pcb: "Hotswap"
weight_material: "Copper"
switches: ["Gateron Oil King"]

# 3. Actual keycap appearance in this photographed build.
profiles: ["Cherry"]
colorways: ["Green", "Cream"]

# 4. Photo/build notes and browsing tags.
rating: "★★★★☆"
lighting: "Window light"
use_case: "Office / writing"
tags: ["favourites", "green", "office"]
```

Include an entry in a combination page when it helps answer one of these questions:

- Can I find this photo by the board, keycap set, colour, profile, switch, form factor, or layout style?
- Does this value describe the exact build in the photos?
- Would this value help compare this build with another build?

Avoid duplicating static master information in combinations. For example, do not repeat `designers`, `kits`, or all possible keyboard `copies` on every combination page unless that information is specifically relevant to the photographed build.

The site automatically creates browse pages for taxonomy terms such as boards, keycap sets, designers, profiles, form factors, layout styles, switches, colourways, makers, and tags.

### Mixed keycap sets on one keyboard

Use one combination page for the actual photographed build, include every set in `keycapsets`, and describe the regions with `keycap_parts`.

Example: GMK Space Cadet alphas with GMK Oblivion mods:

```yaml
title: "Mode Envoy × Space Cadet / Oblivion mix"
boards: ["Mode Envoy"]
keycapsets: ["GMK Space Cadet", "GMK Oblivion"]
profiles: ["Cherry"]
form_factor: "65%"
layout_style: "WK"
tags: ["mixed-keycaps", "grey", "blue"]

keycap_parts:
  - region: "Alphas"
    keycapset: "GMK Space Cadet"
    kits: ["Base"]
  - region: "Modifiers"
    keycapset: "GMK Oblivion"
    kits: ["Git Base", "Hagoromo Cadet Mods"]
  - region: "Accent keys"
    keycapset: "GMK Space Cadet"
    kits: ["Symbols"]
    notes: "Esc and Enter accents."
```

This keeps the photo album in one canonical place while still making it discoverable from both keycap set taxonomy pages.

## 5. Suggested photo naming

Use names that make sense when sorted alphabetically:

```text
cover.jpg
01-front.jpg
02-angle.jpg
03-detail-spacebar.jpg
04-desk-shot.jpg
```

## 6. Why this structure?

Each item is a Hugo page bundle: a folder containing `index.md` and page-specific image files. Hugo can then render the page and its gallery without a database.
